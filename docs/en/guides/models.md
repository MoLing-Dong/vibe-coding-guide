# Choose an Assistant

Different AI coding assistants fit different jobs. Do not choose only by "strongest model." Look at whether the assistant can read your project, edit files, run commands, keep context, and match your workflow.

## Selection Principles

| Scenario | Priority | Recommendation |
| --- | --- | --- |
| Small edits, copy, styling | Speed | Use a fast assistant with smooth editing UX |
| Complex logic or cross-file changes | Reasoning and context | Use an assistant that can inspect the repo, plan, and verify |
| Debugging failed tests | Tool access | Use an assistant that can run commands and search logs |
| Design-to-code work | Multimodal support | Use an assistant that understands screenshots, components, and layout constraints |

## Capability Dimensions

- `Context window`: how much code, requirements, and history it can hold.
- `Tool access`: whether it can search files, edit code, run tests, and inspect browsers.
- `Review quality`: whether it catches edge cases, typing issues, and regressions.
- `Interaction style`: whether it works best as a chat partner or an agent.
- `Cost and speed`: whether it is practical for long pairing sessions.

## Recommended Pairings

| Work stage | How to use the assistant |
| --- | --- |
| Exploring requirements | Ask it to question assumptions, list options, and surface risks |
| Implementing a feature | Ask it to read code, plan, edit in small steps, and verify |
| Troubleshooting | Ask it to reproduce, identify root cause, and then fix |
| Delivery | Ask it to review the diff, add tests, and write notes |

## Choosing Well

- For important code, prefer tools that work directly inside the repository.
- For large tasks, prefer strong planning and review capabilities.
- For frequent small tasks, prefer fast startup and lightweight interaction.
- For private code, confirm your team's data and access policies first.
