# 常见问题排查

Vibe Coding 出问题时，先判断是需求问题、上下文问题、实现问题，还是验证问题。不要急着让 AI “再试一次”，先把失败现象变成可检查的事实。

## 常见问题

| 问题 | 常见原因 | 处理方式 |
| --- | --- | --- |
| AI 改错文件 | 上下文不足或路径不清 | 提供文件路径，让它先读后改 |
| 代码能跑但不符合需求 | 验收标准模糊 | 重写可检查的验收标准 |
| 测试反复失败 | 没有定位根因 | 要求先复现、列假设、再修复 |
| diff 太大 | 任务范围过宽 | 回退或拆成小任务 |
| 风格不一致 | 没提供现有模式 | 指定参考组件或相邻代码 |
| 对话越来越乱 | 上下文漂移 | 新开会话，粘贴交接摘要 |

## 排查模板

```txt
当前现象：
期望结果：
实际结果：
复现步骤：
相关文件：
最近改动：
已经运行的验证命令：
请先分析可能原因，不要直接改代码。
```

## AI 跑偏时怎么拉回

- 复制它的计划，逐条指出哪一步不符合目标。
- 明确“保留哪些改动、撤掉哪些改动”。
- 让它重新总结当前目标和不能碰的范围。
- 必要时让它只输出补丁说明，先不要编辑。

## 验证失败时怎么问

```txt
这个测试失败了。请不要猜。
先解释失败断言在验证什么，再根据相关代码列出 3 个可能原因。
然后选择最可能的原因，给出最小修改方案。
```

排障时最重要的是收敛。每一轮都应该减少不确定性，而不是增加新改动。
