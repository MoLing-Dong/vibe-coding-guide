# 交付上线

Vibe Coding 完成功能只是第一步。真正交付前，需要把 AI 生成的改动变成团队可以理解、验证和维护的代码。

## 上线前检查

- diff 是否只包含本次任务。
- 是否通过测试、类型检查、lint 或构建。
- 是否覆盖空状态、错误状态和权限边界。
- 是否有必要的文档、注释或变更说明。
- 是否清理了调试代码、临时日志和无用文件。
- 是否有回滚方案。

## 让 AI 做交付审查

```txt
请以代码审查者身份检查当前 diff。
重点关注：
1. 是否有无关改动。
2. 是否破坏现有行为。
3. 是否缺少测试或边界处理。
4. 是否有类型、安全或性能风险。
请按严重程度列出问题，不要修改代码。
```

## 写变更说明

交付说明建议包含：

- 做了什么。
- 为什么这么做。
- 怎么验证。
- 有哪些已知风险。
- 是否需要配置、迁移或后续任务。

## 发布后复盘

上线后把结果反馈给 AI，可以让下一次协作更准：

- 哪些提示词有效。
- 哪些上下文缺失导致返工。
- 哪些测试提前发现了问题。
- 哪些步骤可以沉淀为团队模板。

Vibe Coding 的价值会随着流程沉淀不断放大。
