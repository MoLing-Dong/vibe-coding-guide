# AI Debug 工作流

让 AI 修 bug 时，最危险的说法是“你再试试”。进阶 Debug 的核心是先复现、再定位、后修改，每一轮都减少不确定性。

## 1. Debug 的基本原则

Debug 时要遵守四条原则：

- 先读完整错误，再讨论方案。
- 先稳定复现，再修改代码。
- 一次只验证一个假设。
- 修根因，不修表象。

如果 AI 没解释错误就开始改代码，先停下来。

## 2. 提供最小事实

把问题写成事实，而不是情绪。

```txt
当前现象：
点击保存后页面提示成功，但刷新后数据没有变化。

期望结果：
保存后数据持久化，刷新后仍能看到新值。

复现步骤：
1. 打开 /settings/profile。
2. 修改昵称。
3. 点击保存。
4. 刷新页面。

错误信息：
控制台无报错，Network 中 PATCH /api/profile 返回 200。

相关文件：
- src/pages/settings/profile.tsx
- src/api/profile.ts
```

事实越完整，AI 越不容易靠猜。

## 3. 第一轮只分析

```txt
请先分析，不要修改代码。
要求：
1. 解释这个现象可能发生在哪些层。
2. 列出 3 到 5 个可能原因。
3. 说明每个原因如何验证。
4. 推荐优先检查顺序。
```

好的回答应该把问题分层，例如 UI 状态、请求参数、接口返回、缓存、后端持久化。

## 4. 加诊断点

如果问题跨层，不要直接修。先加最小诊断。

```txt
请先只添加临时诊断日志，用来确认：
1. 保存时提交的 payload。
2. 接口返回内容。
3. 成功后本地状态如何更新。

不要改业务逻辑。
```

诊断点要能在下一轮删除。不要把调试日志混进最终提交。

## 5. 验证单个假设

当 AI 选择一个最可能原因后，让它做最小修改。

```txt
现在只验证“保存成功后缓存没有失效”这个假设。
请给出最小修改，不要调整表单结构。
修改后说明如何验证。
```

如果这个假设失败，回到分析阶段，不要继续叠补丁。

## 6. 修复后的回归检查

bug 修好后，至少补一个防回归动作。

- 有测试框架：补一条测试。
- 没测试框架：写清楚手动复现路径。
- UI bug：保留截图或录屏说明。
- 接口 bug：记录请求和响应变化。

给 AI 的提示：

```txt
请基于这个 bug 补一个最小回归检查。
如果不能自动化测试，请给出手动验收步骤。
```

## 7. 常见反模式

| 反模式 | 风险 | 替代做法 |
| --- | --- | --- |
| 让 AI 直接重写模块 | 可能引入更多 bug | 先定位失败层 |
| 一次改多个假设 | 无法判断哪个改动有效 | 一次只测一个假设 |
| 只看最终现象 | 容易修表象 | 追踪数据流 |
| 忽略复现步骤 | 修完无法证明 | 先写复现 |
| 不删除诊断日志 | 污染代码 | 收尾时检查 diff |

## 8. Debug 提示词模板

```txt
你是 Debug 助手。请遵守：
1. 先解释错误和复现路径。
2. 不要直接修改代码。
3. 先列可能原因和验证方式。
4. 选择最可能原因后，只做最小修改。
5. 每次修改后说明如何验证。

当前现象：
期望结果：
复现步骤：
错误信息：
相关文件：
最近改动：
已运行命令：
```

真正的 Debug 不是多试几次，而是每一轮都更接近根因。
