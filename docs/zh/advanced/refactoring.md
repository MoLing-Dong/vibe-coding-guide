# 安全重构

把“帮我重构一下”直接丢给 AI，通常会得到一个很难审查的大 diff。安全重构的核心是先锁定行为，再分步替换结构。

## 1. 什么适合交给 AI 重构

适合：

- 拆分过长函数。
- 消除局部重复逻辑。
- 改善命名和类型。
- 抽出纯函数。
- 迁移一个小组件的内部结构。

不适合一口气交给 AI：

- 重写核心架构。
- 同时改业务逻辑和数据模型。
- 大规模文件移动。
- 没有测试保护的关键支付、权限或删除逻辑。

## 2. 重构前先锁行为

重构不应该改变用户可见行为。先让 AI 总结现有行为。

```txt
请先阅读这个模块，不要修改。
总结：
1. 它对外暴露什么能力。
2. 关键输入输出是什么。
3. 现有边界情况有哪些。
4. 哪些行为必须保持不变。
```

如果它说不清现有行为，就不能开始重构。

## 3. 先补保护

有测试就补测试，没有测试就补手动验收清单。

```txt
在重构前，请先提出最小保护方案：
- 哪些行为需要测试。
- 哪些可以用现有测试覆盖。
- 如果没有测试框架，手动验收步骤是什么。
```

对于风险高的重构，先只提交测试，再做结构调整。

## 4. 分层重构

推荐顺序：

1. 只改命名，不改结构。
2. 抽出纯函数，不改调用方行为。
3. 拆分组件或模块。
4. 删除重复逻辑。
5. 清理旧代码。

每一步都应该能单独通过验证。

## 5. 控制 diff 范围

给 AI 明确边界。

```txt
这次只重构 OrderFilters 内部结构。
不要改接口请求。
不要改页面布局。
不要改样式。
不要移动文件。
每一步结束后运行 npm run typecheck。
```

如果它想顺手修 bug，要先停下来说明：

```txt
如果发现现有 bug，请先记录，不要混入本次重构。
```

## 6. 重构中的检查点

每完成一步，让 AI 回答：

- 对外行为是否保持不变。
- 这一步为什么是纯重构。
- 验证命令是什么。
- diff 里有没有无关改动。

可以使用这个提示：

```txt
请检查刚才这一步是否改变了外部行为。
如果改变了，请指出位置并解释是否必要。
```

## 7. 重构失败时怎么办

如果类型错误、测试失败或 diff 失控，不要继续推进。

处理顺序：

1. 停止新增改动。
2. 让 AI 总结当前 diff。
3. 找出最后一个稳定检查点。
4. 决定保留哪些改动、撤回哪些改动。
5. 把任务重新拆小。

不要让 AI 在混乱状态下继续“修一下”。

## 8. 安全重构模板

```txt
目标：
对 [模块] 做安全重构，提升可读性，不改变外部行为。

边界：
- 不改业务逻辑。
- 不改接口协议。
- 不改 UI 文案。
- 不移动无关文件。

流程：
1. 先总结现有行为。
2. 提出保护方案。
3. 给出最多 5 步重构计划。
4. 每次只执行一步。
5. 每步后运行验证并说明 diff。

验收：
- 现有测试通过。
- 类型检查通过。
- 外部行为保持不变。
```

好的重构应该让代码更清楚，也让审查者更安心。
