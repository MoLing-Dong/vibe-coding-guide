# 完整 AI 编程会话

进阶 Vibe Coding 的关键不是提示词更花哨，而是让一次会话从开始到结束都有节奏：先定目标，再建上下文，然后小步实现、持续验证，最后留下可交接的结果。

## 1. 适用场景

这种会话适合 1 到 3 小时内能完成的真实工程任务，例如：

- 给一个页面增加筛选、排序、空状态或权限提示。
- 修复一个能稳定复现的 bug。
- 给已有接口补一个字段并更新调用方。
- 给缺测试的模块补关键测试。
- 把一段局部代码重构成更清晰的结构。

如果任务需要跨多个系统、改数据库、迁移数据或影响线上配置，先把它拆成多次会话。

## 2. 会话前准备

开工前先准备一个小型任务卡。

```txt
目标：
在订单列表增加退款状态筛选。

业务背景：
运营需要快速筛出待退款订单，减少人工查找。

相关位置：
- src/pages/orders/index.tsx
- src/components/orders/OrderFilters.tsx
- src/api/orders.ts

约束：
- 不改接口协议。
- 筛选值要同步到 URL。
- 复用现有 Select 组件。

验收：
- 默认展示全部订单。
- 选择退款状态后列表刷新。
- 刷新页面后筛选条件保留。
- npm run typecheck 通过。
```

好的任务卡会让 AI 知道什么是成功，也让你知道什么时候该停。

## 3. 第一轮只探索

不要一上来就让 AI 改代码。第一轮只让它读代码、总结现状、列风险。

```txt
请先阅读相关文件，回答：
1. 当前订单列表如何管理筛选状态。
2. URL 查询参数在哪里同步。
3. 需要修改哪些文件。
4. 这次改动的主要风险是什么。

先不要编辑文件。
```

你要检查它是否真的找到了正确位置。如果它开始猜字段、猜组件或扩大范围，先拉回来。

## 4. 第二轮给计划

探索完成后，让 AI 给出 3 到 6 步计划。

```txt
基于刚才的代码阅读，请给出小步修改计划。
每一步都要包含：
- 修改目标
- 涉及文件
- 验证方式

不要超过 6 步。
```

一个好的计划通常长这样：

1. 在筛选类型中增加 `refundStatus`。
2. 在筛选组件中增加下拉选项。
3. 在查询参数构造逻辑中带上该字段。
4. 确认 URL 同步和默认值。
5. 运行类型检查或相关测试。

如果计划同时包含重构、样式重做、接口调整和测试迁移，说明它太大，需要重新拆。

## 5. 第三轮小步实现

让 AI 一次只做一个可验证的小目标。

```txt
先执行第 1 步和第 2 步。
只改筛选类型和 UI，不改接口请求逻辑。
完成后说明改了哪些文件，以及下一步如何验证。
```

每一步结束后看三件事：

- diff 是否只包含本步目标。
- 命名是否复用项目现有风格。
- 下一步是否仍然清楚。

如果 diff 变大，不要继续堆新改动。先要求它解释每个文件为什么必须改。

## 6. 验证失败时停下来

验证失败不是继续乱试的信号，而是进入 Debug 流程的信号。

```txt
typecheck 失败了。
请不要直接修改。
先解释错误信息在说什么，定位相关代码，然后列出 3 个可能原因。
最后选择最可能的原因，给出最小修改方案。
```

进阶协作里，你要避免 AI 用“试试看”覆盖根因分析。

## 7. 收尾与交接

功能完成后，让 AI 做最后一轮交付整理。

```txt
请基于当前 diff 输出交付摘要：
1. 做了什么。
2. 改了哪些文件。
3. 如何验证。
4. 有哪些风险或未覆盖点。
5. 如果我要开 PR，描述应该怎么写。
```

最后你自己再看一次 diff。AI 的总结只能辅助，不能替代人的判断。

## 8. 会话结束模板

每次完成后保留一份短交接，方便下一轮继续。

```txt
本轮目标：
已完成：
修改文件：
验证命令：
未完成：
风险：
下一步建议：
```

这个模板很简单，但它能防止长会话后上下文混乱，也能让下一次 AI 更快进入状态。
