# 上下文工程

上下文工程不是把所有资料都塞给 AI，而是把“完成当前任务所需的信息”压缩成清晰、可靠、可验证的输入。

## 1. 为什么上下文会失效

AI 常见的跑偏原因不是能力不够，而是上下文质量差：

- 只给需求，不给代码位置。
- 给了过期文档，但没有说明状态。
- 粘贴太多无关讨论，冲淡重点。
- 没写验收标准，AI 不知道何时完成。
- 没写禁区，AI 顺手改了不该动的模块。

进阶玩法的目标是让 AI 在动手前拥有一张足够准确的项目地图。

## 2. 上下文包结构

推荐把上下文整理成五块。

```txt
任务：
这次要完成什么。

相关位置：
文件、组件、接口、测试路径。

现有模式：
项目里已经怎么做类似事情。

约束：
不能改什么，必须遵守什么。

验收：
怎样证明完成，运行什么命令。
```

这比一大段自然语言更稳定，因为 AI 可以按字段检查缺口。

## 3. 项目地图

项目地图用于帮助 AI 快速理解仓库，不需要每次都很长。

```txt
项目地图：
- src/pages：页面入口。
- src/components：可复用 UI 组件。
- src/api：接口请求封装。
- src/stores：全局状态。
- tests：单元测试和集成测试。
- npm run typecheck：类型检查。
- npm run test：测试。
```

如果项目有特殊结构，比如 monorepo、插件系统、服务端渲染、数据库迁移，也应该写进去。

## 4. 当前任务上下文

当前任务上下文只保留和这次任务直接相关的信息。

```txt
当前任务上下文：
- 页面：src/pages/orders/index.tsx
- 筛选组件：src/components/orders/OrderFilters.tsx
- 查询参数工具：src/utils/query.ts
- 参考实现：src/pages/customers/index.tsx 里已有状态筛选
- 验证命令：npm run typecheck
```

你可以要求 AI 先自己补全：

```txt
请先搜索和订单筛选相关的文件。
找到后总结现有实现，并告诉我还缺什么上下文。
在我确认前不要编辑。
```

## 5. 禁区和边界

边界写得越清楚，越容易避免无关 diff。

```txt
边界：
- 不改接口返回结构。
- 不新增状态管理库。
- 不重写表格组件。
- 不移动文件。
- 不修改 unrelated 页面。
```

对于遗留项目，还可以加一句：

```txt
如果你认为必须突破边界，先停下来说明原因和替代方案。
```

## 6. 证据优先

不要只让 AI “理解一下项目”。要让它输出证据。

```txt
请用表格列出你找到的相关文件：
| 文件 | 作用 | 为什么相关 | 是否需要修改 |
```

这样你能快速看出它是否找错方向。证据越具体，后续实现越稳。

## 7. 长会话续接

当会话变长，应该主动生成交接摘要。

```txt
请总结当前会话，供下一轮继续：
1. 原始目标。
2. 已确认的上下文。
3. 已做改动。
4. 已运行验证。
5. 当前阻塞点。
6. 下一步建议。
```

如果 AI 开始忘记前文、重复解释、混淆文件名，直接新开会话，把交接摘要贴进去。

## 8. 上下文质量检查

交给 AI 前问自己：

- 有没有明确文件路径。
- 有没有参考实现。
- 有没有禁区。
- 有没有验收方式。
- 有没有说明哪些信息可能过期。
- 有没有要求先读代码再动手。

上下文工程的本质是降低不确定性。每一条上下文都应该帮助 AI 更少猜测。
