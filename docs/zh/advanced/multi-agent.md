# 多角色协作

多角色协作不一定需要真的启动多个 AI 工具。核心是把“计划、实现、审查、验证”拆成不同角色，避免同一个回答同时负责所有判断。

## 1. 为什么要分角色

单角色会话容易出现三个问题：

- 写代码的人倾向于证明自己写得对。
- 计划和实现混在一起，范围容易膨胀。
- 审查不够严格，容易忽略边界。

分角色能制造健康的张力：planner 关注路径，implementer 关注改动，reviewer 关注风险，verifier 关注证据。

## 2. 四个常用角色

| 角色 | 负责 | 不负责 |
| --- | --- | --- |
| Planner | 拆任务、定边界、识别风险 | 直接写代码 |
| Implementer | 按计划小步修改 | 扩大需求 |
| Reviewer | 查 diff、找问题 | 盲目重写 |
| Verifier | 跑命令、读结果、确认验收 | 只看文字总结 |

即使只有一个 AI，也可以按顺序切换角色。

## 3. 单工具模拟多角色

在同一个会话里可以这样做：

```txt
先扮演 Planner。
请只阅读代码并给出计划，不要修改。
```

计划确认后：

```txt
现在切换为 Implementer。
只执行计划第 1 步，完成后停止。
```

实现后：

```txt
现在切换为 Reviewer。
请审查当前 diff，不要修改代码。
```

最后：

```txt
现在切换为 Verifier。
请列出需要运行的验证命令，并解释每个命令证明什么。
```

角色切换能显著减少“边写边圆”的问题。

## 4. 多工具协作

如果你同时使用多个工具，可以这样分工：

- Claude Code：读大仓库、解释复杂业务、做计划。
- Codex CLI：执行具体修改、跑命令、处理补丁。
- ChatGPT 或网页模型：审查 PR 描述、重写文档、做第二意见。

不要让多个工具同时改同一批文件。更稳的方式是串行：一个计划，一个执行，一个审查。

## 5. 交接格式

角色之间要有明确交接。

```txt
交接给 Implementer：
目标：
允许修改：
禁止修改：
执行步骤：
验证命令：
风险：
```

Reviewer 的交接：

```txt
交接给 Reviewer：
原始目标：
验收标准：
当前 diff 摘要：
已运行验证：
需要重点审查：
```

没有交接，多角色只会制造更多上下文混乱。

## 6. 适合多角色的任务

适合：

- 中等规模功能。
- 风险较高的重构。
- 测试失败排查。
- 发布前审查。
- 文档和代码同步更新。

不适合：

- 很小的文案修改。
- 单行配置调整。
- 你自己已经非常确定的简单修复。

多角色是为了降低风险，不是为了增加仪式感。

## 7. 冲突处理

如果 Planner 和 Reviewer 意见冲突，让 AI 给证据。

```txt
Planner 认为需要改 A，Reviewer 认为不应该改。
请分别列出两边的代码证据、风险和推荐决策。
不要直接修改。
```

最终决策仍然由人做。AI 负责把权衡讲清楚。

## 8. 多角色模板

```txt
本任务使用四角色流程：

Planner：
先读代码，输出计划和风险，不改文件。

Implementer：
按确认后的计划逐步修改，每次只做一步。

Reviewer：
审查 diff，按严重程度列问题，不改文件。

Verifier：
运行或建议验证命令，确认结果是否满足验收。

任何角色发现需要扩大范围，都必须先停下来说明原因。
```

分角色的价值，是让 AI 协作更像一个小团队，而不是一个急着交卷的人。
