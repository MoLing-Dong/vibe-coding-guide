# AI 代码审查

AI 不只适合写代码，也适合在提交前扮演严格的 reviewer。关键是让它审查真实 diff，而不是听它泛泛评价“代码质量不错”。

## 1. 什么时候让 AI 审查

建议在这些节点使用 AI Review：

- 功能完成但还没提交。
- diff 超过你能一眼看完的范围。
- 改动涉及权限、金额、数据删除或登录态。
- 测试通过，但你担心边界条件。
- 准备开 PR，需要补描述和风险说明。

不要用 AI Review 替代测试。它擅长发现可疑点，但不能证明代码正确。

## 2. 审查输入

最好的输入是当前 diff、任务目标和验收标准。

```txt
请以代码审查者身份检查当前 diff。

任务目标：
给订单列表增加退款状态筛选。

验收标准：
- 默认显示全部订单。
- 筛选值同步到 URL。
- 刷新后保留筛选状态。

重点关注：
1. 是否有无关改动。
2. 是否破坏现有行为。
3. 是否缺少边界处理。
4. 是否有类型、安全或性能风险。
5. 是否缺少必要测试。

请按严重程度列出问题。不要修改代码。
```

如果工具能直接读取 git diff，让 AI 自己看当前 diff。否则复制核心 diff。

## 3. 严重程度分级

要求 AI 用统一格式输出。

```txt
请按这个格式输出：

严重级别：Blocker / Major / Minor / Nit
位置：文件和行号
问题：
影响：
建议：
```

这样你能区分必须修的问题和可选优化。

## 4. 审查维度

进阶 Review 至少覆盖六类问题。

| 维度 | 关注点 |
| --- | --- |
| 需求一致性 | 是否真的满足验收标准 |
| 行为回归 | 旧功能是否可能被破坏 |
| 边界状态 | 空值、加载、错误、权限 |
| 类型与契约 | 字段名、可选值、接口结构 |
| 安全风险 | 权限绕过、敏感信息、注入 |
| 可维护性 | 无关重构、重复逻辑、命名混乱 |

不要只问“有没有问题”。要明确审查维度。

## 5. 让 AI 审测试

测试也需要被审查。

```txt
请检查新增测试是否真的覆盖了这次需求。
重点看：
1. 是否只验证实现细节。
2. 是否漏掉失败路径。
3. 是否有不稳定等待。
4. 是否没有断言关键行为。
```

很多测试“看起来存在”，但没有保护真实行为。

## 6. 处理审查结果

AI 提出问题后，不要全部照单全收。逐条判断：

- 是否确实和本次目标相关。
- 是否有代码证据。
- 是否必须现在修。
- 是否可以记录为后续任务。

可以继续要求它收敛：

```txt
请把这些发现分成：
1. 本次提交必须修。
2. 可以后续处理。
3. 只是风格建议。
并说明理由。
```

## 7. PR 描述生成

审查完成后，让 AI 基于真实 diff 生成 PR 描述。

```txt
请基于最终 diff 写 PR 描述：

## Summary
- 

## Verification
- 

## Risk
- 
```

PR 描述要短而具体，不要写成宣传稿。

## 8. Review 提示词模板

```txt
请审查当前 diff，不要修改代码。

背景：
目标：
验收标准：
已运行验证：

重点关注：
- 需求是否满足。
- 是否有无关改动。
- 是否可能破坏旧行为。
- 是否缺边界处理。
- 是否缺测试。
- 是否有安全或性能风险。

输出格式：
按 Blocker、Major、Minor、Nit 分组。
每条包含文件位置、问题、影响、建议。
```

AI Review 最有价值的地方，是让你在提交前多一个冷静的检查者。
