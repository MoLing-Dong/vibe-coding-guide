# 任务提示词

好的 Vibe Coding 提示词像一张小型任务卡。它让 AI 知道要做什么、在哪里做、不能碰什么，以及怎样判断完成。

## 基础结构

```txt
背景：为什么要做这件事。
目标：这次具体要完成什么。
上下文：相关文件、接口、组件、错误信息。
约束：不能破坏什么、要遵守什么风格。
验收：怎样证明它完成了。
流程：先计划、等确认、再小步执行。
```

## 示例

```txt
背景：用户反馈报表页加载时没有空状态，容易误以为系统坏了。

目标：给报表页增加空状态提示。

上下文：
- 页面文件：src/pages/reports/index.tsx
- 现有 Loading 和 Error 组件在 src/components/status
- 当 reports.length === 0 且不在加载中时显示空状态。

约束：
- 不改接口请求逻辑。
- 复用现有状态组件风格。
- 先给计划，等我确认后再修改。

验收：
- 有数据时显示原表格。
- 无数据时显示空状态。
- 加载中和错误状态不受影响。
```

## 常用指令

| 指令 | 用途 |
| --- | --- |
| 先阅读相关文件并总结现有实现 | 避免 AI 直接猜代码结构 |
| 给出 2 到 3 个方案和推荐方案 | 适合需求还不确定时 |
| 只改必要文件，不做无关重构 | 控制范围 |
| 每一步结束后说明验证方式 | 保持可检查 |
| 如果信息不足，先问我问题 | 避免硬编造 |

## 提示词质量检查

- 是否能看出成功后的用户体验。
- 是否有明确文件路径或模块边界。
- 是否写清楚不能改什么。
- 是否要求 AI 先计划。
- 是否包含测试或手动验收方式。
