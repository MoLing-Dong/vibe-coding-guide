# 项目上下文

AI 写代码的质量，很大程度取决于它是否理解项目上下文。上下文不是越多越好，而是越相关越好。

## 应该提供什么

| 上下文 | 作用 |
| --- | --- |
| 相关文件路径 | 让 AI 直接定位实现位置 |
| 现有组件或工具函数 | 让 AI 复用项目模式 |
| 数据结构和接口字段 | 避免字段名、类型和状态写错 |
| 错误日志和复现步骤 | 帮助排查问题 |
| 测试命令和构建命令 | 让 AI 自己验证 |
| 团队约定 | 避免风格不一致 |

## 上下文包模板

```txt
项目：React + TypeScript 管理后台。

相关文件：
- src/pages/orders/index.tsx：订单列表页。
- src/components/table/DataTable.tsx：通用表格。
- src/api/orders.ts：订单查询接口。

已有约定：
- 表单使用 react-hook-form。
- 筛选参数同步到 URL。
- 测试使用 npm run test -- orders。

本次任务：
给订单列表增加退款状态筛选。
```

## 让 AI 自己补上下文

在能访问仓库的工具里，可以这样要求：

```txt
请先搜索订单列表、筛选组件和订单接口相关文件。
总结当前实现方式后，告诉我你准备修改哪些文件。
在我确认前不要编辑。
```

## 不要提供什么

- 和任务无关的大段历史讨论。
- 已经过期的设计文档。
- 没有说明用途的整仓文件列表。
- 未解释来源的代码片段。

上下文过多会稀释重点。宁可先少给，再让 AI 说明缺什么。
