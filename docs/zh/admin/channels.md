# 工作流拆解

大任务直接交给 AI，很容易得到一大坨难以审查的改动。更好的方式是把任务拆成可验证的小步骤，让每一步都有明确输入和输出。

## 推荐流程

| 阶段 | 目标 |
| --- | --- |
| 探索 | 找到相关文件和现有模式 |
| 计划 | 明确修改步骤、风险和验证方式 |
| 实现 | 一次只改一个小目标 |
| 验证 | 运行测试、构建或手动检查 |
| 收尾 | 审查 diff、补文档、记录剩余问题 |

## 拆任务示例

原始需求：

```txt
重做订单管理页，让它支持退款筛选、导出和移动端适配。
```

拆成小任务：

1. 阅读订单页，整理当前表格、筛选和接口结构。
2. 增加退款状态筛选并验证 URL 同步。
3. 增加导出按钮，但先接入已有导出能力。
4. 检查移动端布局并只修复明显遮挡。
5. 运行测试和构建，整理剩余风险。

## 给 AI 的控制语句

- 先完成第 1 步，不要编辑文件。
- 每次只提交一个小范围改动。
- 如果发现需要改接口或数据结构，先停下来说明原因。
- 不要顺手重构无关组件。
- 每一步都告诉我怎么验证。

## 何时重新拆分

- AI 给出的计划超过 7 步。
- 单次 diff 涉及太多模块。
- 任务同时包含功能、样式、数据迁移和性能优化。
- 验证方式不清楚。
