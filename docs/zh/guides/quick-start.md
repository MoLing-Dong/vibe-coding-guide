# 快速开始

Vibe Coding 的核心不是“让 AI 随便写代码”，而是用自然语言把目标、上下文、约束和验收标准讲清楚，再让 AI 小步实现、持续验证。第一次练习建议选择一个低风险的小功能。

## 1. 选一个小任务

合适的入门任务通常满足这些条件：

- 能在 30 到 90 分钟内完成。
- 影响范围集中在 1 到 3 个文件。
- 有清晰的成功表现，比如页面多一个筛选项、接口多一个字段、表单多一条校验。
- 失败后容易回滚。

不要一开始就让 AI “重构整个系统”或“从零写完整产品”。先建立节奏，再扩大任务。

## 2. 写第一条任务提示词

把任务写成四段：目标、上下文、约束、验收标准。

```txt
目标：在订单列表页增加“退款状态”筛选。

上下文：
- 当前页面在 src/pages/orders/index.tsx。
- 表格筛选项都放在 OrderFilters 组件里。
- 后端已经返回 refundStatus 字段，可能值是 none、pending、refunded。

约束：
- 保持现有 UI 风格。
- 先给修改计划，等我确认后再改代码。
- 修改后运行相关测试或类型检查。

验收标准：
- 用户可以按退款状态筛选订单。
- 默认显示全部订单。
- 筛选值会同步到 URL 查询参数。
```

## 3. 让 AI 先读代码再计划

第一轮不要急着改。让 AI 先定位相关文件、总结现有模式，并提出 2 到 5 步计划。你要检查：

- 它有没有找到正确文件。
- 它有没有复用现有组件和命名。
- 它有没有提到测试、类型检查或手动验收。
- 它有没有扩大范围。

## 4. 小步执行

确认计划后，让 AI 一次只做一小步。每一步结束后要求它说明：

- 改了哪些文件。
- 为什么这样改。
- 如何验证。
- 是否还有未完成项。

这样你始终知道代码发生了什么，也能在 AI 跑偏时及时拉回来。

## 5. 做收尾检查

完成后不要只看 AI 的文字总结。至少做一次真实验证：

- 运行测试、类型检查或构建命令。
- 打开页面或调用接口做手动检查。
- 查看 diff，确认没有无关重构。
- 让 AI 根据最终代码更新说明或文档。

## 入门检查清单

- 任务是否足够小。
- 上下文是否包含文件路径、现有模式和限制。
- AI 是否先计划再改代码。
- 每一步是否有验证。
- 最终 diff 是否只包含本次目标需要的改动。
