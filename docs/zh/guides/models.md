# 选择助手

不同 AI 编程助手适合不同任务。选择时不要只看“模型最强”，还要看它能否读取项目、编辑文件、运行命令、保留上下文，并适配你的工作节奏。

## 选择原则

| 场景 | 优先级 | 建议 |
| --- | --- | --- |
| 小改动、补文案、写样式 | 速度 | 使用响应快、编辑体验顺的助手 |
| 复杂业务逻辑、跨文件修改 | 推理和上下文 | 使用能读仓库、能规划、能运行验证的助手 |
| 排查测试失败 | 工具能力 | 选择能执行命令、搜索日志、反复验证的助手 |
| 设计稿转页面 | 多模态 | 选择能理解截图、组件结构和布局约束的助手 |

## 常见能力维度

- `上下文窗口`：能一次理解多少代码、需求和历史对话。
- `工具调用`：能否搜索文件、编辑代码、运行测试、查看浏览器。
- `代码审查能力`：能否发现边界条件、类型问题和回归风险。
- `交互方式`：适合聊天式探索，还是适合代理式执行。
- `成本与速度`：是否适合长时间结对开发。

## 推荐搭配

| 工作阶段 | 助手使用方式 |
| --- | --- |
| 探索需求 | 让 AI 提问、列方案、指出风险 |
| 实现功能 | 让 AI 读代码、给计划、小步改文件 |
| 排查问题 | 让 AI 先复现，再定位原因，最后修复 |
| 收尾交付 | 让 AI 做 diff 审查、补测试、写说明 |

## 选择建议

- 重要代码优先选择能直接在仓库内工作的工具。
- 大任务优先选择规划和审查能力强的助手。
- 高频小任务优先选择启动快、交互轻的助手。
- 涉及私有代码时，先确认团队的数据和权限要求。
